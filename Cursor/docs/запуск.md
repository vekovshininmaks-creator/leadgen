## Быстрый старт (минимальный)

### Требования

- Python **3.11+**
- npm опционально — скрипты в `package.json` просто вызывают `py web_app.py`

### Windows (PowerShell)

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\python -m pip install -U pip
.\.venv\Scripts\python -m pip install -r requirements.txt
.\.venv\Scripts\python web_app.py
```

Открыть: `http://127.0.0.1:5173/auth`

Если используете npm:

```powershell
npm.cmd run dev
```

Если PowerShell блокирует `npm.ps1`, вызывайте **`npm.cmd`**.

### Linux / macOS

```bash
python3.11 -m venv .venv
./.venv/bin/pip install -U pip
./.venv/bin/pip install -r requirements.txt
./.venv/bin/python web_app.py
```

Открыть: `http://127.0.0.1:5173/auth`

### Первый вход

- Первый зарегистрированный пользователь получает роль **admin**.
- Telegram и LLM настраиваются в панели: см. [`LLM-настройка.md`](LLM-настройка.md) и раздел про аккаунты в корневом [`README.md`](../README.md).

## Production на VPS (профессионально)

Типовая схема:

- веб-приложение через **gunicorn** + **systemd**
- при необходимости **`telegram_leadgen_bot.py`** отдельным systemd-юнитом (или только из UI — но не дублировать два способа для одной org)
- **nginx** как reverse-proxy и TLS

### Переменные окружения (важно)

| Переменная | Назначение |
|------------|------------|
| `FLASK_SECRET_KEY` | Секрет сессий; в **prod обязателен** |
| `DB_PATH` | Путь к `data.db` |
| `PUBLIC_APP_URL` | Публичный URL панели (биллинг, редиректы) |
| `ENV=prod` / `HTTPS=1` | Secure-cookie — только при **HTTPS** |

Полная таблица и примеры unit-файлов: [`деплой-сервер.md`](деплой-сервер.md).

### Реальный IP и rate limit за nginx

Если gunicorn доступен **только** с `127.0.0.1` через nginx, задайте **`LEADGEN_TRUSTED_PROXY_COUNT=1`** (алиас: `TRUSTED_PROXY_COUNT`). Тогда `request.remote_addr` и лимиты Flask-Limiter учитывают клиента из `X-Forwarded-For`.

**Не включайте** это на установке, где порт приложения открыт в интернет **без** reverse-proxy — злоумышленник сможет подделать заголовки и обходить ограничение по IP.

### Деплой по IP без домена (временно)

Пошаговый ранбук: [`runbook-deploy-by-ip.md`](runbook-deploy-by-ip.md).

**Не включайте** `ENV=prod` и `HTTPS=1`, пока панель открывается по HTTP — иначе cookie не вернутся в браузер и вход сломается.

### Деплой с доменом и TLS

См. разделы systemd + nginx + certbot в [`деплой-сервер.md`](деплой-сервер.md).

## Проверки после запуска

1. **`GET /healthz`** — должен вернуть `"status": "ok"` (и метаданные версии).
2. **502 от nginx** на VPS — часто из-за отсутствия каталогов из `ReadWritePaths` в systemd (шаг 2.5 в [`runbook-deploy-by-ip.md`](runbook-deploy-by-ip.md)).
3. Диагностика UI — глобальный UI debug и выгрузка bundle в браузере.

## Где лежат данные (не перетирать при обновлении кода)

| Путь | Содержимое |
|------|------------|
| `data.db` (или `DB_PATH`) | Пользователи, сессии входа, орги |
| `tenants/org_*` | Конфиг организации, лиды, логи бота, state |
| `sessions/` | Файлы сессий Telethon |

На проде имеет смысл вынести БД и tenant-данные в каталоги вне дерева «чистого» git-клона — см. примеры в [`деплой-сервер.md`](деплой-сервер.md).

## FAQ

**Включили `ENV=prod`, вход по HTTP перестал работать.**  
Ожидаемо: нужен HTTPS или уберите prod/https-флаги до появления TLS.

**`database is locked` у Telethon.**  
Не запускайте параллельно несколько процессов с одной session; не дублируйте бот из UI и из systemd для одной организации.

**Production одной командой (без systemd)** — только для теста:

```bash
gunicorn -w 2 -b 0.0.0.0:5173 wsgi:app
```

Для боевого окружения используйте systemd + nginx из [`деплой-сервер.md`](деплой-сервер.md).
