# Деплой на сервер (Linux)

Кратко: один процесс веб-панели (`web_app.py`) и отдельный процесс бота (`telegram_leadgen_bot.py`). Оба используют один файл SQLite (`DB_PATH`). В этой инструкции рассматривается типовой деплой на VPS Ubuntu/Debian с systemd и nginx-reverse-proxy.

Сводная точка входа по запуску (локально и prod): [`запуск.md`](запуск.md).

> ⚠ **Если у вас пока нет домена / нет HTTPS** — см. раздел [Минимальный сценарий: HTTP-по-IP](#минимальный-сценарий-http-по-ip). По HTTP cookie должны идти **без** флага `Secure`, иначе пользователь не сможет залогиниться.

---

## Переменные окружения

| Переменная | Назначение | Пример |
|---|---|---|
| `FLASK_SECRET_KEY` | Секрет сессий Flask. **Обязателен в prod.** Сгенерировать: `python3 -c "import secrets;print(secrets.token_hex(32))"` | `2f4e...` |
| `DB_PATH` | Путь к `data.db`. Если не задан — рядом с приложением. | `/var/lib/leadgen/data.db` |
| `ORG_ID` | Идентификатор организации для бота (из таблицы `orgs`). | `1` |
| `PUBLIC_APP_URL` | Публичный URL панели **без** завершающего слэша (для `return_url` ЮKassa). | `https://leadgen.example.com` или `http://203.0.113.10` |
| `YOOKASSA_SHOP_ID` / `YOOKASSA_SECRET_KEY` | Учётные данные ЮKassa. URL вебхука: `<PUBLIC_APP_URL>/api/billing/webhook/yookassa`. | — |
| `ENV` | `prod` включает строгие проверки и **`Secure` cookie** (требует HTTPS). | `prod` |
| `HTTPS` | Альтернативно: `1`/`true` принудительно делает cookie `Secure`. | `1` |
| `LEADGEN_WEBLEADS_WORKER` | `0` отключает фоновый воркер «Лидов по сайтам» в этом процессе. | `1` |
| `LEADGEN_TRUSTED_PROXY_COUNT` | Доверять `X-Forwarded-*` от N слоёв reverse-proxy (обычно **1** за nginx). **0** по умолчанию — безопасно для прямого доступа к gunicorn. Не ставить >0 без прокси. | `1` |

> При `ENV=prod` или `HTTPS=1` cookie выставляется как `Secure`, и браузер не отправит её по HTTP. Поэтому при HTTP-по-IP **не ставьте** эти флаги (см. ниже).

## Подготовка ОС (Ubuntu/Debian)

```bash
sudo apt update
sudo apt install -y python3 python3-venv python3-pip nginx git
sudo useradd -r -s /usr/sbin/nologin leadgen   # системный пользователь сервиса
sudo mkdir -p /opt/leadgen /var/lib/leadgen /var/backups/leadgen
sudo chown -R leadgen:leadgen /opt/leadgen /var/lib/leadgen /var/backups/leadgen
```

```bash
sudo -u leadgen git clone <repo> /opt/leadgen/app
sudo -u leadgen python3 -m venv /opt/leadgen/venv
sudo -u leadgen /opt/leadgen/venv/bin/pip install -U pip
sudo -u leadgen /opt/leadgen/venv/bin/pip install -r /opt/leadgen/app/requirements.txt
```

Перед первым запуском `systemd` создайте каталоги из `ReadWritePaths` (иначе возможен сбой монтирования namespace и **502**):

```bash
sudo mkdir -p /opt/leadgen/app/logs /opt/leadgen/app/tenants /opt/leadgen/app/sessions /opt/leadgen/app/data
sudo chown -R leadgen:leadgen /opt/leadgen/app/logs /opt/leadgen/app/tenants /opt/leadgen/app/sessions /opt/leadgen/app/data
```

> На версиях Python <3.11 поставьте `python3.11` через `deadsnakes` PPA. Минимум — Python **3.11**.

## Запуск веб-панели через gunicorn

`/etc/systemd/system/leadgen-web.service`:

```ini
[Unit]
Description=Leadgen web console
After=network.target

[Service]
Type=simple
User=leadgen
Group=leadgen
WorkingDirectory=/opt/leadgen/app
Environment=FLASK_SECRET_KEY=__сгенерируйте_секрет__
Environment=DB_PATH=/var/lib/leadgen/data.db
Environment=PUBLIC_APP_URL=https://leadgen.example.com
Environment=YOOKASSA_SHOP_ID=
Environment=YOOKASSA_SECRET_KEY=
Environment=ENV=prod
Environment=HTTPS=1
Environment=LEADGEN_TRUSTED_PROXY_COUNT=1
ExecStart=/opt/leadgen/venv/bin/gunicorn -w 2 -b 127.0.0.1:8000 web_app:app
Restart=always
RestartSec=3

# Безопасность процесса
NoNewPrivileges=yes
ProtectSystem=strict
ReadWritePaths=/var/lib/leadgen /opt/leadgen/app/tenants /opt/leadgen/app/sessions /opt/leadgen/app/data /opt/leadgen/app/logs
PrivateTmp=yes

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now leadgen-web
sudo systemctl status leadgen-web --no-pager
```

## Запуск Telegram-бота отдельным сервисом

В UI бот можно стартовать кнопкой «Старт» — он запустится как дочерний процесс веба. Для prod удобнее вынести его в отдельный systemd-юнит (бот переживёт рестарты веба).

`/etc/systemd/system/leadgen-bot.service`:

```ini
[Unit]
Description=Leadgen Telegram bot
After=network.target

[Service]
Type=simple
User=leadgen
Group=leadgen
WorkingDirectory=/opt/leadgen/app
Environment=DB_PATH=/var/lib/leadgen/data.db
Environment=ORG_ID=1
ExecStart=/opt/leadgen/venv/bin/python3 telegram_leadgen_bot.py --org-id ${ORG_ID} --data-db ${DB_PATH}
Restart=always
RestartSec=5

NoNewPrivileges=yes
ProtectSystem=strict
ReadWritePaths=/var/lib/leadgen /opt/leadgen/app/tenants /opt/leadgen/app/sessions /opt/leadgen/app/data /opt/leadgen/app/logs
PrivateTmp=yes

[Install]
WantedBy=multi-user.target
```

> Не запускайте бот через UI и через systemd одновременно для одной org — будут два процесса. Перед включением `leadgen-bot` создайте те же каталоги, что для веб-юнита (см. блок `mkdir` выше). Для управления только из UI оставьте `leadgen-bot.service` отключённым.

## Nginx (с TLS)

```nginx
server {
    listen 443 ssl http2;
    server_name leadgen.example.com;
    ssl_certificate     /etc/letsencrypt/live/leadgen.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/leadgen.example.com/privkey.pem;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    client_max_body_size 5m;

    location /static/ {
        alias /opt/leadgen/app/static/;
        expires 7d;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 90s;
    }
}

server {
    listen 80;
    server_name leadgen.example.com;
    return 301 https://$host$request_uri;
}
```

Сертификат: `sudo apt install -y certbot python3-certbot-nginx && sudo certbot --nginx -d leadgen.example.com`. Cron-обновление сертификата ставит сам certbot.

## Минимальный сценарий: HTTP-по-IP

Применяется, когда домена ещё нет, а развернуться нужно прямо сейчас. **Это временный режим** — без HTTPS пароли и сессии передаются по сети открытым текстом. Перейдите на HTTPS как только появится домен.

### 1. Не ставьте `ENV=prod` и `HTTPS=1`

В `leadgen-web.service` уберите две строки:

```ini
# Environment=ENV=prod
# Environment=HTTPS=1
```

Иначе cookie `auth_token` уйдёт со флагом `Secure` и не вернётся по HTTP — пользователь не сможет войти.

`FLASK_SECRET_KEY` всё равно установите — без него сессии работать не будут.

### 2. nginx без TLS

`/etc/nginx/sites-available/leadgen-http.conf`:

```nginx
server {
    listen 80 default_server;
    server_name _;

    client_max_body_size 5m;

    location /static/ {
        alias /opt/leadgen/app/static/;
        expires 7d;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

```bash
sudo ln -s /etc/nginx/sites-available/leadgen-http.conf /etc/nginx/sites-enabled/leadgen-http.conf
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx
```

### 3. Откройте порт 80

```bash
sudo ufw allow 80/tcp     # если ufw активен
```

### 4. Зайдите по IP

`http://<публичный-IP>/auth` — окно входа. Адрес панели: `http://<IP>/`.

### 5. Поставьте напоминание перейти на HTTPS

Когда привяжете домен, выполните: `sudo certbot --nginx -d ваш-домен`, затем верните `ENV=prod`/`HTTPS=1` в `leadgen-web.service` и `sudo systemctl restart leadgen-web`.

## Резервное копирование (cron)

Скрипт `/opt/leadgen/backup.sh`:

```bash
#!/bin/bash
set -e
TS=$(date +%F-%H%M)
DEST=/var/backups/leadgen
mkdir -p "$DEST"
tar czf "$DEST/leadgen-$TS.tar.gz" \
    -C / var/lib/leadgen \
    -C /opt/leadgen/app tenants sessions data 2>/dev/null || true
find "$DEST" -name 'leadgen-*.tar.gz' -mtime +14 -delete
```

```bash
sudo chmod +x /opt/leadgen/backup.sh
sudo crontab -e
# добавьте:
# 0 3 * * * /opt/leadgen/backup.sh
```

Что попадает в архив: `data.db` (если хранится в `/var/lib/leadgen/`), tenant-конфиги (`tenants/`), Telegram-сессии (`sessions/`), аватары (`data/avatars`).

## Цикл обновления приложения

Шпаргалка «с ПК на сервер» (git push / pull, что не трогать в БД): [`синхронизация-и-домен.md`](синхронизация-и-домен.md).

```bash
cd /opt/leadgen/app
sudo -u leadgen git pull
sudo -u leadgen /opt/leadgen/venv/bin/pip install -r requirements.txt
sudo systemctl restart leadgen-web
# при отдельном systemd-боте:
sudo systemctl restart leadgen-bot
```

Миграции `data.db` идемпотентны: при старте `web_app` создаёт недостающие таблицы (`outreach_queue`, `calls`, `payments`, `conversations`, `bot_runs`, `web_leads`, `web_lead_jobs`) и индексы.

### Боевая БД vs копия с ноутбука

- Если в `leadgen-web.service` задан `DB_PATH=/var/lib/leadgen/data.db`, **именно этот файл** — продовая база (пользователи, org, оплаты).
- Файл `data.db` в каталоге приложения **`/opt/leadgen/app`** при обычном копировании проекта с ПК **не заменяет** `DB_PATH`, если последний указывает на `/var/lib/leadgen/`.
- При выкладке обновлений: обновляйте **код** (`git pull` / синхронизация файлов), **не подменяйте** `/var/lib/leadgen/data.db` локальной копией без бэкапа и осознанного переноса.

### 502 Bad Gateway и `status=226/NAMESPACE`

Симптом: nginx открывается, страница **502**; `systemctl status leadgen-web` не **active**. В журнале: **Failed to set up mount namespacing** или **226/NAMESPACE**.

1. Убедитесь, что существуют все пути из `ReadWritePaths`, в первую очередь **`/opt/leadgen/app/logs`** (и при необходимости `tenants`, `sessions`, `data`).
2. `sudo chown -R leadgen:leadgen /opt/leadgen/app/logs ...`
3. `sudo systemctl restart leadgen-web`.

Запасной вариант: временно убрать из юнита `ProtectSystem=strict`, `ReadWritePaths=...`, `PrivateTmp=yes`, выполнить `daemon-reload` и `restart`.

Проверка приложения: `curl -sS http://127.0.0.1:8000/healthz` на сервере.

## Лидоген по сайтам

Дополнительный модуль (страница «Воронка → Лиды по сайтам»). Поиск по выдаче через SerpAPI + парсер контактов на странице компании.

Что нужно:

- В `requirements.txt` уже есть `httpx>=0.27` — устанавливается стандартно.
- Получите API-ключ на [serpapi.com](https://serpapi.com/).
- Откройте страницу «Лиды по сайтам» → раздел «Подключения» → вставьте ключ → «Сохранить».
- Запросы и импорт CSV-доменов кладутся в `web_lead_jobs`. Их обрабатывает фоновый daemon-thread, который стартует автоматически при импорте `web_app.py` (по одному на каждый gunicorn worker).
- Чтобы отключить парсер на конкретном инстансе (например, в воркере, обслуживающем только webhook’и) — переменная `LEADGEN_WEBLEADS_WORKER=0`.
- Найденный сайт можно одной кнопкой «В CRM» — он попадёт в `conversations` со `status='active'` и `lead_user_id='web:<домен>'`.

> Парсер сайтов **не** ходит на `localhost`/приватные IP (защита от SSRF), уважает `Content-Type: text/html` и ограничивает каждую страницу 1.5 MB / 12 секундами.

## Оплата (ЮKassa)

Клиент платит сумму тарифа без отдельной комиссии в интерфейсе. После успешной оплаты ЮKassa шлёт уведомление на `<PUBLIC_APP_URL>/api/billing/webhook/yookassa`. Сервер перепроверяет платёж через API ЮKassa и обновляет подписку. Эндпоинт уже защищён rate-limit + проверкой `Content-Type` + аудит-логом (`login_audit`).

## Чек-лист после деплоя

1. `sudo systemctl status leadgen-web` → `active (running)`.
2. Открыть `<PUBLIC_APP_URL>/auth` → возможность регистрации.
3. Зарегистрировать первого пользователя; в БД он автоматически становится владельцем `org_id=1` (роль `client`/`admin`).
4. На странице «Настройки и лимиты» → «Подключения» внести LLM-ключ (опционально) и SerpAPI-ключ (для Лидов по сайтам).
5. На странице «Контакты (бот)» подключить Telegram-аккаунт (api_id/api_hash из my.telegram.org).
6. Запустить бота из UI или через systemd-юнит. Убедиться, что таблица `bot_runs` обновляется (heartbeat) — на странице «Бот» прогресс-бар не залипает.
7. Проверить cron-бэкап: `/opt/leadgen/backup.sh` руками → должен появиться файл в `/var/backups/leadgen/`.
