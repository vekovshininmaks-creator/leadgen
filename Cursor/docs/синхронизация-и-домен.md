# Быстрая синхронизация локальной и серверной версии + бесплатное имя для HTTPS

Короткая шпаргалка. Полный деплой: [`деплой-сервер.md`](деплой-сервер.md). Развёртывание с нуля по IP: [`runbook-deploy-by-ip.md`](runbook-deploy-by-ip.md).

---

## 1. Как устроена синхронизация

| Что | Где «источник правды» |
|-----|------------------------|
| **Код** (Python, шаблоны, `static/`) | Обычно **ваш ПК** → коммит в git → на сервере `git pull` |
| **Боевая база** `data.db` | **Сервер** (`DB_PATH`, часто `/var/lib/leadgen/data.db`). Локальный `data.db` с ноутбука **не подменяет** прод автоматически и не должен заливаться «на autopilot» |

Идея: вы меняете код локально, отправляете в репозиторий, на VPS подтягиваете те же коммиты и перезапускаете сервисы.

---

## 2. Быстрый цикл: ПК → сервер (рекомендуется, через git)

### На своём компьютере

1. Убедитесь, что изменения сохранены и проходят вашу проверку.
2. Зафиксируйте и отправьте в удалённый репозиторий (имя ветки подставьте свою, чаще `main` или `master`):

```bash
git add -A
git status
git commit -m "Кратко: что сделано"
git push origin main
```

Если сервер клонирует **другую** ветку — пушьте в неё или на сервере переключите ветку один раз (`git checkout …`).

### На сервере (SSH под `root` или пользователем с `sudo`)

Один блок команд — обновить код, зависимости и процессы:

```bash
sudo -u leadgen bash -lc 'cd /opt/leadgen/app && git pull && /opt/leadgen/venv/bin/pip install -r requirements.txt'
sudo systemctl restart leadgen-web
sudo systemctl restart leadgen-bot
```

- Строку с **`leadgen-bot` оставьте**, если бот запущен **отдельным** systemd-юнитом. Если бот только из веб-интерфейса — эта команда не обязательна (или сервис отключён — тогда `restart` просто скажет, что юнит неактивен).
- Если **`requirements.txt` не менялся**, этап `pip install` почти мгновенный; пропускать его можно, но привычка «всегда pip после pull» избавляет от сюрпризов после обновлений.

Проверка:

```bash
curl -sS http://127.0.0.1:8000/healthz
sudo systemctl status leadgen-web --no-pager
```

Снаружи откройте панель в браузере (по IP, домену или sslip — как у вас настроено).

## 3. Если git на сервере не используете

Синхронизируйте каталог приложения **инструментами копирования** (с ПК на сервер), например `rsync` или WinSCP, в **`/opt/leadgen/app`**, не затирая **`/var/lib/leadgen/data.db`**.

Пример с Linux/macOS (подставьте пользователя, хост и путь к локальной копии проекта):

```bash
rsync -avz --delete --exclude '.venv' --exclude 'data.db' --exclude '.git' \
  ./путь/к/локальному/leadgen/ root@ВАШ_IP:/opt/leadgen/app/
```

После копирования на сервере всё равно выполните `pip install -r requirements.txt` от пользователя `leadgen` и `systemctl restart` как выше.

---

## 4. Обратное направление: забрать **бэкап** с сервера на ПК

Продовая БД обычно **не** «синхронизируют» в git. Если нужна копия для отладки:

1. На сервере сделайте снимок (или используйте готовый cron из [`деплой-сервер.md`](деплой-сервер.md)).
2. Скопируйте архив или `data.db` **только вручную** и храните вне репозитория.

Никогда не заливайте локальный `data.db` поверх прода без резервной копии на сервере и понимания, что перезапишете пользователей и оплаты.

## 5. Бесплатное «доменное имя» для панели и Let's Encrypt

На «голый» IP браузерный **доверенный** HTTPS от Let's Encrypt **не выдают**. Нужен **hostname**, который резолвится в ваш VPS. Варианты без покупки зоны:

### 5.1. Самый быстрый: sslip.io (или nip.io)

Сервис сам выдаёт имя из вашего IP. **Точки в IP заменяются на дефисы** + суффикс `.sslip.io`.

**Пример:** IP `203.0.113.10` → имя `203-0-113-10.sslip.io`.

Проверка:

```bash
getent hosts 203-0-113-10.sslip.io
```

Дальше: nginx с этим `server_name`, `certbot --nginx -d 203-0-113-10.sslip.io`, в systemd **`PUBLIC_APP_URL=https://…`**, **`HTTPS=1`**, при необходимости **`ENV=prod`**.

Пошагово с готовыми блоками команд: раздел **S1** в [`runbook-deploy-by-ip.md`](runbook-deploy-by-ip.md) (там же пример конфига и certbot).

### 5.2. Бесплатный поддомен с регистрацией аккаунта

Удобно, если нужно «читаемое» имя, а не цепочка цифр:

| Сервис | Комментарий |
|--------|-------------|
| [DuckDNS](https://www.duckdns.org/) | Поддомен вида `ваш-ник.duckdns.org`, указываете IP в панели |
| [FreeDNS (afraid.org)](https://freedns.afraid.org/) | Много зон на выбор, нужна регистрация |
| Другие динамические DNS | Аналогично: создали A-запись на IP сервера |

После того как имя стабильно ведёт на ваш VPS: тот же сценарий — **nginx** + **`certbot --nginx -d ваше-имя`** + обновление **`PUBLIC_APP_URL`** (как в [`деплой-сервер.md`](деплой-сервер.md)).

### 5.3. Что проверить в облаке

Для HTTPS откройте **входящие TCP 80 и 443** в панели провайдера VPS (не только `ufw` на машине). Иначе certbot и браузеры не достучатся.

---

## 6. Мини-чеклист после синхронизации

1. `git pull` (или rsync) выполнен без ошибок.
2. `pip install -r requirements.txt` от пользователя `leadgen`.
3. `systemctl restart leadgen-web` (и бот — если отдельным юнитом).
4. `/healthz` с localhost сервера отвечает `ok`.
5. В браузере открывается `/auth`, вход работает; при HTTPS включены `HTTPS=1` / `ENV=prod` согласно [`деплой-сервер.md`](деплой-сервер.md).
